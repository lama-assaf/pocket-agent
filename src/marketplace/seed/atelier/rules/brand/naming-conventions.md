# naming conventions

how things get named inside the product and the company.

## products

- short. 1-2 words.
- pronounceable on first read.
- searchable (the brand name comes up first when you google it).
- available in the channels that matter (domain, npm, app store).

## features

- describe what the feature does, not what it's built with.
- in the user's words, not the team's.
- consistent within a product family (don't have "spaces" in one product and "workspaces" in another for the same concept).

## internal projects

- code names are fine internally. don't leak them to users.
- avoid code names that imply the feature's purpose to outsiders.

## variables and entities

- the word the user sees should match the word in the api and the database.
- if users call them "deals" and the database calls them "opportunities", the gap shows up in every customer interaction.

## what to avoid

- compound words that need explanation
- spelling tricks ("Xeed" instead of "Seed")
- names that work in english but mean something embarrassing in another major language
- names that are very close to an established brand in the same category
- names you have to defend in every meeting
